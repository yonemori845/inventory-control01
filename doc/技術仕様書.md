## 1. 技術スタック

### 1.1 フロントエンド

| 技術 | 役割・備考 |
|------|------------|
| **Next.js**（App Router） | UI・Route Handlers・Server Actions |
| **React** | コンポーネント |
| **TypeScript** | 全体の型 |
| **Tailwind CSS** | スタイリング |
| **shadcn/ui**（Radix ベース） | コンポーネント primitives |
| **@supabase/ssr** / **@supabase/supabase-js** | 認証・DB・Storage クライアント |
| **PWA** | `next-pwa` または **Serwist**、または手動 SW（**7章**） |

バーコード読取：**`@zxing/browser`** 等を採用してもよいが、**現リポジトリの実装**（2025-03-21 時点）はブラウザ標準の **`BarcodeDetector`（Shape Detection API）** を使用する。対応ブラウザ外では手入力にフォールバックする。ZXing 等へ統一する場合は本節と `doc/実装状況とドキュメント照合.md` を更新すること。

### 1.2 バックエンド・BaaS

| 技術 | 役割 |
|------|------|
| **Supabase** | Auth、PostgreSQL、Storage、（将来）RLS |
| **PostgreSQL** | 正規データ（ホストは Supabase。開発初期のみ Docker 可） |

アプリ専用の**自前 API サーバーは置かない**（Next.js + Supabase で完結）を既定とする。

### 1.3 品質・テスト

| 技術 | 役割 |
|------|------|
| **ESLint** / **Prettier**（任意） | 静的解析・整形 |
| **Vitest** または **Jest** | 単体テスト（採用をリポジトリで 1 つに） |
| **Playwright** | E2E（要件：注文確定と在庫減の検証を CI に含める） |

### 1.4 バージョン固定方針

- **Node.js**：本書 **§2.1** の範囲で `.nvmrc` または `engines` に固定。
- **依存パッケージ**：**lockfile**（`pnpm-lock.yaml` / `package-lock.json`）をコミットする。
- メジャーアップデートはセキュリティ・Next の要件がある場合に限定し、本書の改訂履歴に残す。

## 2. 実行環境

### 2.1 Node.js

- **推奨：20.x LTS**（メンテナンス LTS）。
- **最低**：Next.js が要求する最小 Node に従う（プロジェクト初期化後、`engines` に明記）。

### 2.2 パッケージマネージャ

- **推奨：`pnpm`**（`corepack enable` で統一）。
- **代替**：`npm` でも可。リポジトリ内では **1 種類に統一**し、README に記載。

### 2.3 推奨開発環境（任意）

- VS Code / Cursor + ESLint 拡張。
- Supabase CLI（ローカル DB や migration 用）。

## 3. ローカル開発

### 3.1 リポジトリの取得と起動手順（案）

1. Node を §2.1 に合わせる。  
2. `pnpm install`（または npm install）。  
3. `.env.local` を `.env.example` からコピーし、**§5** の変数を設定。  
4. `pnpm dev` で Next.js 起動（ポートは既定 3000）。  

※ リポジトリ未初期化の場合は、初期化後に本節を**実際のコマンド**に差し替える。

### 3.2 Docker（PostgreSQL）の役割と寿命

| 項目 | 内容 |
|------|------|
| 目的 | スキーマ検討の初期のみ、**PostgreSQL をローカルで起動**する（要件：早期 Supabase 移行）。 |
| 構成案 | `docker-compose.yml` で `postgres:16` 等 1 サービス。 |
| 接続 | `DATABASE_URL=postgresql://...` を `.env.local` に設定。 |
| 寿命 | **スキーマが安定したらクラウド Supabase を主**とし、Docker は任意に降格。 |

ORM を使わず **Supabase クライアント + SQL migration** のみでもよい。その場合 `DATABASE_URL` は migration CLI 専用。

### 3.3 Supabase ローカル／クラウドの切り替え

| モード | 用途 |
|--------|------|
| **クラウド Supabase（推奨・本番同等）** | Auth・Storage・RLS 検証。`.env.local` にプロジェクトの URL／キー。 |
| **Supabase CLI `supabase start`**（任意） | 完全ローカルスタック。チーム方針で採用可否を決める。 |

**切り替えルール**：`NEXT_PUBLIC_SUPABASE_URL` と `NEXT_PUBLIC_SUPABASE_ANON_KEY`（およびサーバー用キー）を**環境ごとに差し替える**。コードに URL を直書きしない。

## 4. Supabase

### 4.1 プロジェクト構成

| リソース | 用途 |
|----------|------|
| **Authentication** | Email、Google プロバイダ |
| **Database** | 業務テーブル（`機能設計書` §6） |
| **Storage** | 商品画像バケット（例：`product-images`） |
| **Edge Functions** | 必須ではない。必要になったら本書に追記 |

Dashboard で **Project URL**、**anon public key**、**service_role**（秘密）を取得する。

### 4.2 Auth（Google／メール）

| 設定 | 内容 |
|------|------|
| Email | Supabase Auth の Email プロバイダを有効化。確認メールは開発時は無効化可。 |
| Google | Google Cloud で OAuth クライアント作成。**承認リダイレクト URI** に Supabase のコールバック URL を登録。 |
| サイト URL | Supabase の「Site URL」を Vercel の本番 URL（およびローカル `http://localhost:3000`）に合わせる。 |
| Redirect URLs | プレビュー用 `*.vercel.app` を許可するかはチームで決定（許可すると設定が増える）。 |

**profiles 作成**：Auth トリガー（`on auth.users insert`）またはアプリ初回ログインで `profiles` に UPSERT（`機能設計書` §4.1）。

### 4.3 PostgreSQL・マイグレーション

- **マイグレーション**：`supabase/migrations/*.sql` を **Supabase CLI** または Dashboard SQL で適用。
- **型生成**：`supabase gen types typescript` で `src/types/database.ts` 等へ出力（パスは `リポジトリ構造定義書` に合わせる）。

#### 4.3.1 注文確定と DB トランザクション（RPC）

- **要件**：注文ヘッダ・明細・在庫減算・`inventory_movements` を **同一トランザクション**とする（`要件定義書` §8.2）。
- **推奨実装**：PostgreSQL の **関数（ストアドプロシージャ）** を 1 本定義し（例：`place_order`）、その中で在庫検証→INSERT/UPDATE→COMMIT／失敗時 ROLLBACK を行う。
- **アプリ側**：Server Action または Route Handler が **service_role またはユーザーセッション付きで `rpc('place_order', …)` を 1 回呼ぶ**形にすると、クライアントからの複数リクエスト分割を避けられる。
- **権限**：RPC は `SECURITY DEFINER` とする場合、**監査と最小権限**に注意（必要なら専用ロール）。

### 4.4 Row Level Security（第1段階／第2段階）

| フェーズ | 方針 |
|----------|------|
| **第1段階** | **認証済みユーザーに広めの SELECT/INSERT/UPDATE**、または開発中は一部テーブルで RLS 無効（**本番デプロイ前に見直し**）。要件：実質オール管理者。 |
| **第2段階** | `profiles.role` に応じたポリシー、管理者専用テーブル／操作の制限。 |

**原則**：`anon` キーはクライアントに埋め込まれるため、**RLS 無効のまま本番にしない**。第1段階でも「`auth.uid()` が null なら拒否」程度は推奨。

### 4.5 Storage（商品画像）

| 項目 | 案 |
|------|-----|
| バケット名 | `product-images`（公開読み取り）または非公開＋署名 URL |
| パス規約（推奨の単一参照） | **`doc/商品画像_収集と利用方針.md` §5**（`group_code` やジャンルセグメントを含む階層の例、`seed/...` との併用）。**論理上の正は親商品（商品グループ）単位**（要件定義書 §8.3）。 |
| パス規約（技術的代替） | `sku/{sku_id}/...` のように SKU 主キーでキーを切る構成も可能。運用時は**同一 `product_group` の SKU が同じオブジェクトを指す**ようパスまたは値を揃え、重複アップロードを避けること。 |
| ポリシー | 認証ユーザーのアップロード可、読み取りは公開または認証のみ（デザインに合わせて決定） |

### 4.6 管理者画面（ユーザー一覧）と service_role

- **SCR-ADM-USERS**（`機能設計書`）：Auth のユーザー一覧取得・メール表示などは、通常 **`anon` キーでは足りない**。
- **方針**：当該処理のみ **サーバー側**で **`SUPABASE_SERVICE_ROLE_KEY`** を用いた Supabase Auth Admin API（または同等）を呼ぶ。キーは **環境変数・サーバーモジュールに限定**し、クライアントに露出しない（§10）。
- **代替**：Edge Function + 限定 IP 等は本課題では必須としない。

## 5. 環境変数

### 5.1 一覧（公開可／秘密）

| 変数名 | 公開 | 必須 | 用途 |
|--------|------|------|------|
| `NEXT_PUBLIC_SUPABASE_URL` | 可（ブラウザに露出） | ○ | Supabase API URL |
| `NEXT_PUBLIC_SUPABASE_ANON_KEY` | 可 | ○ | クライアント用 anon key |
| `SUPABASE_SERVICE_ROLE_KEY` | **秘密**（サーバーのみ） | △ | 管理 API・バイパス RLS が必要な処理のみ。原則最小限 |
| `DATABASE_URL` | 秘密 | △ | Docker／migration 専用。Supabase の「接続文字列」と別物に注意 |
| `NEXT_PUBLIC_APP_URL` | 可 | △ | 絶対 URL が必要なリンク（OGP、メール等） |
| `NEXT_PUBLIC_DEFAULT_TAX_RATE` | 可 | △ | 例：`0.1`（機能設計の固定税率と一致） |
| `TZ` またはアプリ内定数 | - | △ | 集計の「本日」：**`Asia/Tokyo`** を推奨（機能設計書 §1.3） |
| `NEXT_PUBLIC_SENTRY_DSN` | 可 | △ | クライアントエラー送信 |
| `SENTRY_AUTH_TOKEN` | 秘密 | △ | CI でのソースマップアップロード等 |
| `NEXT_PUBLIC_VERCEL_ANALYTICS_ID` | 可 | △ | Vercel が注入する場合あり（実装に従う） |
| `PLAYWRIGHT_TEST_BASE_URL` | 秘密（CI） | △ | E2E 用（例：`http://127.0.0.1:3000`） |
| `ALLOW_DEV_SEED` | 秘密 | △ | **`true` のときのみ**シード API／Server Action を許可する。**シード可否の正は本変数**とする（`NODE_ENV` だけに依存しない。Preview も `production` になりうるため）。 |

**禁止**：`SUPABASE_SERVICE_ROLE_KEY` を `NEXT_PUBLIC_*` にしない。

**補足**：ローカル開発では `.env.local` に `ALLOW_DEV_SEED=true` を入れ、CI・本番・Preview では**未定義または `false`**。

### 5.2 `.env.example` との対応

- リポジトリ直下に **`.env.example`** を置き、上表のキーを**値なしまたはダミー**で列挙。
- `.env.local` は **gitignore** 固定。

### 5.3 Vercel 上の設定

| 環境 | 設定 |
|------|------|
| **Production** | 本番 Supabase の URL／anon／（必要なら）サーバー用シークレット |
| **Preview** | 別 Supabase プロジェクトを推奨、または同一プロジェクトでプレフィックス分離（運用で選択） |
| **Pull Request** | Preview と同様 |

`ALLOW_DEV_SEED` は **Production / Preview に設定しない**。

## 6. Vercel

### 6.1 プロジェクト設定

- Framework Preset：**Next.js**。
- Root Directory：リポジトリがモノレポでなければ空。
- Build Command：`pnpm build`（または `npm run build`）。
- Output：Next.js 既定。

### 6.2 Preview / Production

- **main**（または **production ブランチ**）マージで Production デプロイ（**§9** と連動）。
- **その他ブランチ／PR** は Preview。E2E は Preview URL またはローカルビルドで実行するかをワークフローで固定。

### 6.3 デプロイフックと GitHub の関係

| パターン | 内容 |
|----------|------|
| **A. Vercel Git 連携** | push で Vercel が自動ビルド。CI は GitHub Actions でテストのみ。 |
| **B. Actions からデプロイ** | `vercel deploy` を **E2E 成功後**に実行（要件：テスト後デプロイ）。 |

要件（**テスト成功後のみデプロイ**）を満たすには **B**、または **A + 「デプロイ前に必須チェック」相当**の運用を文書化する。本課題では **B を推奨**（再現性が高い）。

## 7. PWA

### 7.1 Web App Manifest

- `app/manifest.ts`（Next Metadata API）または `public/manifest.webmanifest`。
- **name**、**short_name**、**icons**（512／192 等）、**start_url**、`display: standalone` または `browser`。

### 7.2 Service Worker 戦略

- **precache（静的）**：アプリシェル（レイアウトに必要な HTML／JS／CSS）、アイコン、オフラインフォールバック用ページ。
- **キャッシュしない（原則 network-only）**：**アプリの API Route**、**Supabase への fetch**、**個人・業務データ**（要件：**オフライン中の在庫・注文の編集・同期は行わない**）。

**文言の注意**：「Network First」は *キャッシュあり* の戦略であり、**業務 API を誤ってキャッシュしない**こと。データ取得系は **fetch ハンドラで該当 URL をキャッシュ対象外**にするか、`network-only` 相当にする。シェル用静的資産のみ precache と明記する。

### 7.3 オフライン範囲（要件との一致）

| 要件（要件定義 §7.2） | 技術的対応 |
|------------------------|------------|
| シェル・ナビ・静的アプリがオフラインで開ける | precache に layout CSS/JS、フォールバック HTML |
| 業務データのオフライン編集・同期キューなし | fetch ハンドラで API はキャッシュしない／オフライン時はエラー画面 |

## 9. CI/CD（GitHub Actions）

### 9.1 ワークフロー一覧（案）

| ワークフロー名 | トリガー | ジョブ内容 |
|----------------|----------|------------|
| `ci.yml` | `pull_request`（`main` 向け）、`push`（`main`） | install、lint、typecheck、unit test、**E2E**、（任意）build |
| `deploy.yml` | `workflow_run`（ci 成功後）または `push` to `main` with `needs: ci` | **E2E 成功後**のみ Vercel デプロイ（§6.3 パターン B） |

単一ワークフローに **jobs: lint → test → e2e → deploy** と `needs` チェーンでまとめてもよい。

### 9.2 E2E 実行タイミング

- **PR ごと**に実行し、**失敗時はマージ不可**（§9.4）。
- E2E 前に `pnpm build && pnpm start` で本番相当サーバーを起動し、`PLAYWRIGHT_TEST_BASE_URL` を合わせる。
- **ログインシナリオ**：CI では **メール／パスワード**のテストユーザーを使用する（`機能設計書` §11.1）。Google OAuth の E2E は必須としない。シークレットにテスト用パスワードを置く場合は GitHub Actions の **Secrets** を使用。

### 9.3 デプロイ条件（成功時のみ）

- **lint / typecheck / unit（あれば）/ E2E** がすべて成功した後にのみデプロイ job を実行する。
- いずれか失敗時：**デプロイ job をスキップ**。

### 9.4 ブランチ保護チェックリスト

- [ ] `main` に **直接 push 禁止**（または許可する場合は例外理由を README に記載）。
- [ ] **Required status checks** に CI ワークフロー名（例：`ci / e2e`）を指定。
- [ ] **Require branches to be up to date** を推奨。
- [ ] Vercel の **Ignored Build Step** を CI 任せにする場合は、二重デプロイに注意（§6.3）。

## 10. セキュリティ

### 10.1 キー・トークンの取り扱い

- **service_role** は GitHub Secrets・Vercel Environment（Production のみ必要なら Production のみ）に保存。
- ローテーション手順は Supabase Dashboard から再発行。

### 10.2 クライアントに載せてはいけないもの

- `SUPABASE_SERVICE_ROLE_KEY`
- 任意の **私用 API キー**、**DB の root パスワード**

クライアントバンドルに含まれるかは **Next.js の `server-only` 境界**でレビューする。
