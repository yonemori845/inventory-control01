## 3. データ・サーバー処理と設計の対応

### 3.1 マイグレーション（`supabase/migrations/`）

| 内容 | 設計参照 | 実装 |
|------|----------|------|
| スキーマ（profiles, product_*, inventory_movements, orders, order_lines） | 機能設計 §6 | ✅ `20250321120000_initial_schema.sql` |
| profiles 自動作成 | 機能設計 §4.1、技術 §4.2 | ✅ `20250321130000_profiles_on_auth_user.sql` |
| `adjust_sku_quantity`, `barcode_inbound`, `import_product_csv_rows` | 機能設計 §4.4–4.5, §6.5 | ✅ `20250321140000_inventory_rpcs.sql` |
| `inventory_movements.reason = csv_import` | 機能設計 §6.5 | ✅ CSV RPC 内で記録 |
| 注文確定の単一トランザクション RPC | 機能設計 §6.4、技術 §4.3.1 | ❌ **未定義**（orders テーブルはあるが確定処理なし） |

### 3.2 RLS

- 初期スキーマで **RLS 有効**＋`authenticated` に広めのポリシー。技術仕様書「第1段階」方針と一致。
- **anon** は業務テーブルにアクセスできない想定（ミドルウェアで未ログインを `/login` へ）。

### 3.3 Server Actions の配置

- 設計では `src/modules/<domain>/actions.ts` または `src/app/(app)/.../actions.ts` が「案」。
- **実装**: `src/app/actions/inventory.ts` に在庫系を集約。リポジトリ構造定義書に実パスを追記済み（§3.3）。

## 6. 不具合・リスク候補（コードレビューベース）

1. **注文・ダッシュボード未実装** … 要件のコアの半分が未着手。在庫のみでは「一元管理」のデモが弱い。
2. **BarcodeDetector** … Safari / 一部環境で未対応。フォールバック文言はあるが、E2E で環境固定が必要。
3. **型定義の手書き** … RPC 引数名とクライアントのずれはビルドで検知しにくい。`gen types` 推奨。
4. **service_role 未使用の管理者画面** … 設計どおり実装する際はキー管理を技術仕様 §10 に従うこと。
5. **要件 §2.3** … GitHub Actions・E2E・デプロイパイプラインがないため、課題の「完成の定義」には未到達。
