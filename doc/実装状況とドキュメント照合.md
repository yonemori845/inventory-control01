## 6. 不具合・リスク候補（コードレビューベース）

1. ~~**注文・ダッシュボード**~~ … フェーズ D で実装済み。**Supabase に `place_order` マイグレーション未適用**の環境では注文確定が失敗する点に注意。
2. **バーコードスキャン** … ネイティブ未対応時は ZXing にフォールバック。E2E ではカメラモックまたは手入力経路の固定が必要。
3. **型定義の手書き** … RPC 引数名とクライアントのずれはビルドで検知しにくい。`gen types` 推奨。
4. **`reference_type` / `reference_id`** … `place_order` で `order_sale` 行に設定済み（§3.2）。将来の拡張で**別 reason にも紐づけを広げる**場合は、一覧 UI と整合させること。
5. ~~**税・端数の二重定義**~~ … `pricing.ts` と `place_order` SQL で同一ルールを固定済み。変更時は両方を同期すること。
6. **service_role 未使用の管理者画面** … 設計どおり実装する際はキー管理を技術仕様 §10 に従うこと。
7. ~~**要件 §2.3** … CI/E2E 不在~~ … **フェーズ F でワークフロー・typecheck・Playwright を整備済み**。残るのは **Vercel 本番への環境変数**・**ブランチ保護の有効化**など**運用側の最終確認**（`task.md` 参照）。
8. ~~**`npm run typecheck` 不在**~~ … **`package.json` に追加済み**。CI でも実行される。
9. **商品画像の DB 理想形** … 方針は **`product_groups` に集約**だが、スキーマは現状 `product_skus.image_path` のみ。移行前は**同一グループで同一パス**を入れる運用で表示を揃える（`doc/商品画像_収集と利用方針.md`）。
10. **並行実装の巻き戻し** … コミットに含めないまま試作したコードを消したり、`git reset` したりすると、**ドキュメントや会話の記憶だけが先に進んだ状態**になりやすい。**対策はコミットまたは本書・`task.md` の即時更新**（下記 §9）。

## 9. 層が出そうな箇所（これからの負債ポイント）

実装が進むと、ここで**責務が重なりやすい**（デバッグ難・二重実装の温床）。

| 箇所 | リスク |
|------|--------|
| **注文確定** | クライアント状態・Server Action・RPC・在庫・税計算が一度に絡む。**RPC 1 本＋薄い Action**が設計どおり。ここを分割せずに JS で複数 `insert` するとトランザクションが崩れる。 |
| **集計（ダッシュ・レポート）** | `orders` / `order_lines` / `product_skus` の JOIN と日付境界（Tokyo）。**同じクエリをコピペ**すると期間オフバイワンが二重発生。`lib/report-data` 等の**読み取り専用モジュール 1 箇所**に寄せることを推奨。 |
| **CSV 入出力** | ヘッダー・型・エラーメッセージが `csv.ts` と RPC の JSON 期待値で**ズレやすい**。機能設計 §8 と実装の列順を表で固定する。 |
| **在庫アラート** | `alerts.ts` と画面ごとの閾値表示が分岐すると「一覧ではアラートなし・詳細ではアラート」等が起きうる。**判定関数は 1 箇所**（現状は `isInventoryAlert` に集約されているが、D 以降も維持）。 |
