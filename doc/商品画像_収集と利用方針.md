本書は、**画像素材の集め方・利用上の判断・データ上の理想形**を記録する。実装手順や API 仕様そのものは `技術仕様書`・`機能設計書` を正とし、本書と矛盾する場合は**実装後に各所を更新**する。

---

## 1. 目的と範囲

### 1.1 目的

- 在庫一覧・SKU 詳細などで、**商品を視覚的に識別**できるようにする。
- 身内での閲覧およびプレゼンにおいて、**説明可能で見栄えのよい**素材選定と記録を行う。
- アプリを**先に拡張**し、将来的に**他ジャンルへ載せ替え可能なひな形**としても再利用できるよう、方針と命名の考え方を残す。

### 1.2 範囲

| 扱う | 扱わない（別文書・別フェーズ） |
|------|-------------------------------|
| 素材の入手方針、著作権・出典の扱い | 画像アップロード UI の詳細仕様（画面 ID 単位は機能設計書） |
| Storage 上のオブジェクトの命名・階層の**推奨** | RLS・バケットポリシーの具体 SQL（技術仕様書） |
| DB 上の「正」の所在の**理想形**と現状スキーマの差分の記録 | マイグレーションの実行結果 |

### 1.3 利用シーンとリスク許容（合意）

- **主な利用シーン**：身内での共有、プレゼンでの提示。
- **著作権の考え方**：コンプライアンス文書レベルの厳密さまでは求めない一方、**商用利用で訴訟に発展しうるような使い方は避ける**。説明に耐え、気持ちよく見せられるラインを満たすことを目標とする。

## 2. システム上の扱い（現状）

### 2.1 保管

- **Supabase Storage** の公開バケット（例：`product-images`）にオブジェクトを配置する想定。詳細は `doc/技術仕様書.md` §4.5。

### 2.2 技術仕様書 §4.5 との関係

- **バケット名・公開／非公開・ポリシー**などインフラ寄りの案は `doc/技術仕様書.md` §4.5 を正とする。
- **オブジェクトキー（パス）の推奨階層・論理単位（親商品）**は**本書 §4・§5**を正とする。技術仕様に SKU id ベースのパス例がある場合は**技術的代替**であり、運用上は同一 `product_group` の SKU で**同じオブジェクト**を指すように揃えること。

### 2.3 アプリケーションでの参照

- `product_skus.image_path` に、次のいずれかを格納する設計である。
  - **公開バケット相対パス**（例：`product-images/...`）— アプリは `NEXT_PUBLIC_SUPABASE_URL` と組み合わせて公開 URL を組み立てる。
  - **http(s) の絶対 URL** — そのまま表示に利用可能。
- SKU 詳細画面（`/inventory/sku/[skuId]`）では `next/image` により表示する。未設定時はプレースホルダ文言を表示する。
- 在庫一覧では、現時点で画像枠はプレースホルダに近い状態であり、**将来の表示拡張**の余地がある。

### 2.4 表示上の制約（選定時の注意）

- 詳細画面では**正方形に近い領域**で **`object-cover`（中央基準のトリミング）** が使われる。
- 素材選定では、**被写体をフレーム中央に置き、周囲に余白**がある写真を優先すると切れにくい。
- UI 上の案内として、**JPEG / PNG / WebP、最大 5MB** を想定している（実装・文言は変更されうる）。

## 3. 画像に関する要件（数・表現・カバー）

| 項目 | 内容 |
|------|------|
| 枚数規模 | **数十枚**を目安とする |
| トーン | **リアル写真**を優先する（実写・実物感） |
| 想定ジャンル | **衣類・アパレル**（検索キーワード・シードの雰囲気のたたき） |
| カバー率 | シード上の商品について、**可能な限りすべて**に画像が付く状態を目標とする（グループ単位で見ると**全グループに 1 枚**に近いイメージ） |

## 4. 確定方針（意思決定の整理）

以下は、検討の結果として**方針として固定**した内容である。

### 4.1 画像の論理単位

- **親商品（商品グループ）単位**で画像を管理するのが正とする。
- 同一 `product_group` に属する SKU（色・サイズ違い）は、**原則同一画像を共有**する。

### 4.2 データモデル上の「正」と現状実装の差分

| 区分 | 内容 |
|------|------|
| **理想（本書での正）** | 画像パスは **`product_groups`（親商品）側**に保持し、SKU はグループを参照して表示する。 |
| **現状（2026-04-05 時点のスキーマ）** | `image_path` は **`product_skus` の列**としてのみ存在する（`product_groups` には画像列なし）。 |
| **移行方針** | 実装フェーズで、`product_groups` への列追加（または同等の参照）と、一覧・詳細の取得クエリの整理を行う。**本書は理想形を記録**し、マイグレーション完了後に「現状」欄を更新する。 |
| **移行までの運用（任意）** | 同一グループ内の全 SKU に**同一の `image_path` 文字列**を複製しておくことで、画面上はグループ単位と同等の見え方にできる。 |

### 4.3 素材ソース

- **主ソース**：ライセンスがサイト上で確認できる**ストック写真**（例：Unsplash、Pexels 等）。
- **補助**：自撮り・身内での撮影（ストックで揃いきらない場合）。

### 4.4 見た目の統一感

- **同じ作家・同じ照明にまで揃えない**（選定コストを抑える）。
- **雰囲気がそこそこ揃えばよい**レベルを目標とする。

### 4.5 出典・ライセンスの記録

- **README** または**プレゼン資料**に、利用したストック元を**一行〜短い節**でまとめて記載する（サイト名と、利用にあたっての考え方が伝わる程度でよい）。

### 4.6 推奨しない入手方法

- 出所・利用条件が不明な、検索結果の**無断転載**に依存する方法（リスクだけでなく、プレゼン時の**説明**もしづらい）。

## 5. Storage・ファイル運用の推奨

実装の詳細は技術仕様に従う。素材整理のため、次を**推奨**とする。

### 5.1 オブジェクトキー（パス）の考え方

- **グループコードや意味のある識別子**がパスに含まれると、CSV・シード・手動更新と対応付けしやすい。
- 将来、**ジャンルやテナント**でフォルダを分けられる構造（例：`product-images/apparel/<group_code>/main.webp`）にすると、他ジャンルへの載せ替えがしやすい。

### 5.2 解像度の目安

- 詳細の正方形表示であれば、**一辺 800〜1200px 程度**あれば多くの場合十分である（5MB 制限内で収まることを確認する）。

---

## 6. 将来・汎用化（ひな形としての位置づけ）

- 本リポジトリは**アパレル想定の具体デモ**として画像セットを用意するが、**アプリのデータモデルと Storage の命名規則**は、別ジャンルに差し替えても破綻しにくい形を意識する。
- 画像の選定・枚数・ジャンルは**後から差し替え可能**なアセットであり、**方針本文（§4・§5）**をひな形として残す。
