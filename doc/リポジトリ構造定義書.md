# 在庫管理システム リポジトリ構造定義書

| 項目 | 内容 |
|------|------|
| 文書名 | 在庫管理システム リポジトリ構造定義書 |
| 版数 | 0.3.1 |
| 作成日 | 2025-03-21 |
| 根拠 | `doc/要件定義書.md` v1.8、`doc/機能設計書.md` v0.3.1 |
| 整備計画 | `doc/スペック駆動開発_ドキュメント整備計画.md` |

本書は **Next.js（App Router）＋ Supabase** を前提としたディレクトリの**責務と配置ルール**を定義する。実際のリポジトリが未作成の場合、**初期化時にこの構造を採用する**。

---

## 1. 目的・適用範囲

### 1.1 目的

- ファイルの置き場所の議論を減らし、**機能設計の画面・ドメイン**とパスを対応させる。
- 秘密情報・サーバー専用コードを**クライアントバンドルから隔離**する。

### 1.2 適用範囲

- **単一 Next.js アプリ**のリポジトリ（モノレポではない）を想定する。

---

## 2. ルートディレクトリ一覧

| パス | 責務 |
|------|------|
| `doc/` | 要件・設計・計画（本書含む） |
| `src/` | アプリケーション本体（**推奨**：`src/app` 構成） |
| `public/` | 静的ファイル、PWA manifest 連携用アイコン等 |
| `supabase/` | Supabase CLI 用（`migrations/`, `config.toml` 等）※導入時 |
| `e2e/` | Playwright（または任意 E2E）の spec・fixtures |
| `.github/workflows/` | CI/CD ワークフロー |

**注**：`src/` を使わない `app/` ルート直下構成にする場合は、下表の `src/app` を `app` に読み替える（**どちらか一方に統一**し、採用を技術仕様書に記載）。

---

## 3. アプリケーション配下（`src/`）

### 3.1 ルーティング（App Router）

| パス | 対応画面 ID（機能設計 §3） |
|------|-----------------------------|
| `src/app/login/page.tsx` | SCR-AUTH-LOGIN |
| `src/app/(app)/page.tsx` | SCR-DASH |
| `src/app/(app)/inventory/page.tsx` | SCR-INV-LIST |
| `src/app/(app)/inventory/sku/[skuId]/page.tsx` | SCR-INV-SKU |
| `src/app/(app)/inventory/movements/page.tsx` | SCR-INV-HIST |
| `src/app/(app)/orders/page.tsx` | SCR-ORD-LIST |
| `src/app/(app)/orders/new/page.tsx` | SCR-ORD-NEW |
| `src/app/(app)/orders/[orderId]/page.tsx` | SCR-ORD-DET |
| `src/app/(app)/reports/page.tsx` | SCR-RPT |
| `src/app/(app)/settings/page.tsx` | SCR-SET |
| `src/app/(app)/admin/users/page.tsx` | SCR-ADM-USERS |

`(app)` は**認証済みレイアウト**用の Route Group（名称は任意だが 1 つに統一）。

### 3.2 レイアウト・ルートグループ

- `src/app/layout.tsx` … ルート HTML、プロバイダ、フォント。
- `src/app/(app)/layout.tsx` … サイドナビ／トップバー、認証ガード（ミドルウェアと併用可）。

### 3.3 API Routes / Server Actions の置き場方針

| 種別 | 置き場（案） |
|------|----------------|
| Route Handler | `src/app/api/.../route.ts`（CSV、ヘルス、エクスポート用バイナリ等） |
| Server Actions | **ドメインごとの** `src/modules/<domain>/actions.ts` または `src/app/(app)/.../actions.ts`（プロジェクトで統一） |

**実装メモ（2025-03-21）**：在庫ドメインの Server Actions は **`src/app/actions/inventory.ts`** に集約されている。今後モジュール分割する場合は本表の「案」に寄せ、本書を更新する。

**原則**：認可が必要な更新はサーバー側のみ。サービスロールキーは **サーバーモジュールのみ**。

## 4. ドメイン・モジュール配置方針

### 4.1 採用パターン

- **機能（ドメイン）別**を第一選択とする：`inventory` / `orders` / `reports` / `settings` / `admin`。
- 各モジュールに **queries（読み取り）**、**actions（更新）**、**types** をまとめてもよい。

### 4.2 在庫・注文・レポートの境界

| ドメイン | 責務 |
|----------|------|
| `inventory` | product_groups / product_skus / inventory_movements、CSV、スキャン入庫 |
| `orders` | カート相当のクライアント状態は UI 層、確定トランザクションはサーバー |
| `reports` | 集計クエリ、エクスポート生成 |
| `dashboard` | SCR-DASH 専用の集計（`reports` に寄せてもよい） |

---

## 5. 共有ライブラリ

### 5.1 `src/lib/` の役割

| サブパス（案） | 責務 |
|----------------|------|
| `lib/supabase/server.ts` | サーバー用クライアント作成 |
| `lib/supabase/client.ts` | ブラウザ用クライアント |
| `lib/utils.ts` | cn() 等 |
| `lib/constants.ts` | 税率、日付フォーマット等 |

### 5.2 Supabase クライアント

- **クライアント**：anon key のみ。RLS 前提。
- **サーバー**：ユーザーセッション付きクライアント。管理操作用は別クライアント（サービスロール）を**限定ファイル**に閉じる。

### 5.3 型定義

- `src/types/database.ts` … Supabase 生成型（`gen types`）。
- ドメイン型は `src/modules/*/types.ts` または `src/types/`。

---

## 6. テストコード

### 6.1 単体・結合

- `src/**/*.test.ts` または `__tests__/`（**1 方式に統一**）。

### 6.2 E2E（Playwright 等）

- `e2e/*.spec.ts`
- 画面 ID または `data-testid` を **機能設計 §11** のシナリオと対応させる。

### 6.3 命名規則

- spec 名：`e2e-order-confirm.spec.ts` のように**機能英語スネーク**または **kebab** で統一。

---

## 7. シード・マイグレーション

### 7.1 SQL／スクリプトの置き場

- `supabase/migrations/*.sql` … スキーマ変更。
- `supabase/seed.sql` または `scripts/seed-dev.ts` … 開発用 append シード（**本番で実行しない**）。

### 7.2 実行手順

- `技術仕様書.md` に CLI コマンドと環境変数を記載する。

---

## 8. 静的アセット・PWA

### 8.1 `public/` のルール

- ファビコン、`icons/`、公開する静的画像のみ。機密は置かない。

### 8.2 manifest・アイコン

- `public/manifest.webmanifest`（または Next Metadata API で生成）。
- アイコン複数解像度を `public/icons/` に配置。

---

## 9. 禁止事項・例外手続き

### 9.1 禁止事項

- ブラウザに **サービスロールキー**を渡すコード・`.env` のプレフィックス誤り。
- `app/` または `src/app/` 以外への任意の Route Handler 散乱（**`src/app/api` に集約**）。

### 9.2 構造変更時の手続き

- ディレクトリの責務変更時は**本書を先に更新**し、PR 説明にリンクする。

## 10. 機能設計との対応

| 機能領域 | 主な配置パス（案） |
|----------|---------------------|
| SCR-AUTH-LOGIN | `src/app/login/`, `src/modules/auth/` |

| SCR-RPT | `src/app/(app)/reports/`, `src/modules/reports/` |
