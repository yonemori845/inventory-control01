## 1. 文書の目的と範囲

本書は、Web ベースの在庫・注文（売上）一元管理システムについて、実装・テスト・デプロイの判断に使えるよう機能要件・非機能要件・データ要件を定義する。

本書は**設計書・実装仕様書ではない**。API の詳細、画面ワイヤー、DB の物理名確定は別文書または実装フェーズで扱う。

下位仕様・運用仕様は次を参照する（スペック駆動開発の永続ドキュメント）。

- `doc/スペック駆動開発_ドキュメント整備計画.md` … 文書間の関係と執筆順（**1.5**：テコ入れ運用 §5.1）  
- `doc/機能設計書.md` … 画面・データ・API・例外・テスト観点  
- `doc/技術仕様書.md` … スタック、環境変数、Supabase／Vercel、CI/CD、PWA、監視  
- `doc/リポジトリ構造定義書.md` … ディレクトリ責務と配置ルール  
- `doc/デザイン設計書.md` … UI トークン、コンポーネント、画面別デザイン  
- `doc/task.md` … 実装のフェーズ別チェックリスト（エージェント作業単位。**1.3**：テコ入れ手順）

## 2. 背景・目的

### 2.1 背景

小売・倉庫業務において、商品在庫と販売（注文）情報を分断せず扱い、バーコードを用いて入力作業を減らす必要がある。本課題では架空データにより、その業務イメージを再現する。

### 2.2 目的

- **在庫管理**と**注文（売上）管理**を同一システム上で一元化する。
- **JAN／バーコードスキャン**により、入庫登録と注文明細追加を効率化する。
- **ダッシュボードとレポート**により、売上・在庫の状況を可視化し、経営判断のイメージを支援する。
- **モダンな Web 技術スタック**、**CI/CD**、**PWA**、**監視**を課題要件として満たす。

### 2.3 成功基準（完成の定義）

- **本番相当のデプロイ**（Vercel）上でアプリケーションが動作し、**Supabase（PostgreSQL／Auth／Storage）に接続**していること。
- **GitHub Actions** により E2E テスト実行後、**成功時のみ Vercel へデプロイ**され、**テスト失敗時はマージがブロック**されるブランチ保護が前提として機能していること（運用上のセットアップを含む）。

## 3. 用語定義

| 用語 | 定義 |
|------|------|
| 商品（親） | ユーザーが認識する同一商品のまとまり。色・サイズは子 SKU に分ける。 |
| SKU | 在庫・JAN・バーコード操作の最小単位。親商品に属する。 |
| 入庫 | 在庫数量を増やす操作。本システムのバーコードスキャンは**入庫のみ**を対象とする。 |
| 注文 | 明細行を持つ販売記録。**注文の確定は売上計上とみなす**。 |
| 正規データ | 業務データの唯一の根拠（Single Source of Truth）。本システムでは **Supabase 上の PostgreSQL** とする。 |

## 4. システム概要

### 4.1 概要構成

- **クライアント**：Web ブラウザ（レスポンシブ）。**PWA** としてインストール可能。
- **アプリケーション**：ホスティングは **Vercel** を想定。フレームワークは `plan.txt` に記載の **Next.js / React / TypeScript** を主とする。
- **バックエンド**：**Supabase**（PostgreSQL、認証、Storage）。開発初期は **Docker 上の PostgreSQL** でスキーマを固め、**早期に Supabase に移行**する。
- **監視**：**Sentry**（エラー）、**Vercel Analytics**（アクセス解析）。

### 4.2 データ連携方針

- **Google スプレッドシートを正とする連携は採用しない**（`plan.txt` 後半の記述は旧案とし、本書では対象外）。
- 参考データとして `doc/template_DB.ods`（存在する場合）をシード設計の参考とする。ODS の列名は実装時に本書のエンティティへ対応付ける。

## 5. 利用者・権限

### 5.1 認証

- **Google アカウントによるログイン**を提供する。
- **メールアドレスとパスワード**によるログインを提供する（デモ用アカウント配布を想定）。

### 5.2 権限（フェーズ区分）

| フェーズ | 内容 |
|----------|------|
| 第1段階（本課題の主スコープ） | **実質すべての利用者を管理者相当**として扱う。機能への厳格な制限は最小とする。 |
| 第2段階（拡張） | **管理者／一般ユーザー**の分離、**Row Level Security（RLS）** の細分化、**監査ログ**の強化を行う。 |

### 5.3 管理者向け画面

- ユーザー一覧・権限管理の**画面・データ項目**は第1段階から用意してよい。アクセス制御の厳格化は第2段階で効かせる。

## 6. 機能要件

### 6.1 認証・セッション

- ログイン／ログアウト、セッション維持。
- 未認証ユーザーは業務画面にアクセスできない（方針は実装で統一）。

### 6.2 ダッシュボード

次を表示する。

- 売上推移、期間売上、期間注文数、平均注文額。
- 売上ランキング上位 **TOP10**。
- **在庫アラート**（在庫が少ない SKU。判定は **6.3.7** のルールに従う）。

### 6.3 在庫管理

#### 6.3.1 サマリ・一覧

- 商品グループ数、総 SKU 数、総在庫数、在庫アラート件数などのサマリ。
- **同一親商品の色・サイズ違いは入れ子（階層）表示**する。
- 一覧の **SKU コード** から **SKU 詳細**（`/inventory/sku/[skuId]`）へ遷移できる。

#### 6.3.2 検索

- 商品名、商品コード、JAN による検索。

#### 6.3.3 CSV による商品情報更新

- 在庫一覧の **「CSV ダウンロード」** ボタンで、所定ヘッダー付きの CSV を取得できる。
- アップロードにより商品マスタ（親子に相当する情報）を更新できる。

#### 6.3.4 在庫数量の手動更新

- 画面上の在庫数量変更がデータベースに反映される。
- **在庫変動履歴**（**6.3.6** およびデータ要件の `inventory_movements`）に記録する。

#### 6.3.5 バーコード（JAN）スキャンによる入庫

- JAN をスキャンした SKU の在庫数量を**加算**し、DB を更新する。
- **入庫以外の用途（出庫スキャン等）は本要件に含めない。**

#### 6.3.6 入出庫履歴

- 入庫・手動更新・注文による減算など、在庫に影響した履歴を**閲覧**できる（理由区分・日時・SKU 等）。
- 履歴行の **SKU** から SKU 詳細へ遷移できる。

#### 6.3.7 在庫予測・発注推奨アラート

- **ルールベースのみ**とする（例：現在庫が再発注点以下、または安全在庫を下回る）。
- 統計モデル・機械学習による予測はスコープに含めない。

### 6.4 注文管理（売上の入力）

#### 6.4.1 サマリ

- 総注文数、本日の注文、総売上（税込）、平均注文額など（表示項目は `plan.txt` に準拠しつつ実装で調整可）。

#### 6.4.2 注文の作成

- バーコードスキャンにより明細に商品（SKU）を追加できる。
- 明細数量は増減（＋／－）できる。
- 合計金額を表示する。
- **注文確認（確定）**により注文一覧に反映し、DB に保存する。

#### 6.4.3 注文詳細

- 注文番号などから詳細を表示する。
- **税込・税別の内訳を表示**できること。税法レベルの厳密な計算は求めない。

#### 6.4.4 注文と売上の関係

- **注文の確定時点で売上として計上**する。
- **キャンセル・返品による売上・在庫の巻き戻しは行わない**（シンプルなモデル）。

#### 6.4.5 注文確定と在庫の整合

- **注文確定時に、明細の数量分だけ該当 SKU の在庫を減算する。**
- 減算は **在庫変動履歴**に記録する。
- **在庫不足がある場合は注文全体を確定しない**（トランザクションをロールバックし、ユーザーに不足 SKU を示す）。**マイナス在庫は許容しない**。

#### 6.4.6 決済

- **決済機能は不要**（外部決済連携は行わない）。

### 6.5 レポート・分析

- 売上データの分析、比較、チャート表示（日別売上、商品別ランキング、期間比較など。`plan.txt` の③④に準拠）。
- **PDF および Excel 形式でのエクスポート**。

### 6.6 設定

- アカウント情報の表示・必要に応じた更新。
- **データベース接続状況**の表示（Supabase への疎通確認など）。
- 権限情報の表示（第1段階ではロールの表示に留めてよい）。
- **開発環境のみ**：ダミーデータの**追加投入**（後述 8.3）。

### 6.7 管理者機能

- ユーザー一覧、権限（ロール）の変更 UI（第1段階では全員管理者相当でもよい）。

### 6.8 エラー監視・ログ（運用要件としての機能）

- **Sentry** によりエラーを自動収集する。
- **Vercel Analytics** によりアクセスを解析可能とする。

### 6.9 CI/CD（リポジトリ・パイプライン要件）

- **GitHub Actions** で E2E テストを実行する。
- テスト成功時のみ **Vercel へデプロイ**する。
- テスト失敗時は **マージをブロック**する（ブランチ保護ルールと連動）。

## 7. 非機能要件

### 7.1 ユーザビリティ

- **レスポンシブデザイン**により、複数画面幅で利用可能とする。

### 7.2 PWA

- **Progressive Web Application** として、インストール（ホーム画面追加）に対応する。
- **オフライン時**：アプリの**シェル（レイアウト・ナビ）および静的アセット**が開けること。**オフライン中の在庫・注文データの編集や同期キューは行わない**。オンライン復帰後は通常利用とする。

### 7.3 バーコード

- カメラ等を用いた **JAN／バーコード読取**を在庫入庫および注文作成のいずれかで利用する（`plan.txt` に準拠）。

### 7.4 セキュリティ（第1段階）

- 認証は Supabase Auth に委ねる。
- **RLS の厳格な業務ルールは第2段階**とするが、本番 Supabase 利用時は**セキュリティ推奨設定**（ anon key の取り扱い等）は実装ガイドに従う。

### 7.5 テスト

- E2E テストにおいて、**注文確定後に在庫が減少していること**を、在庫十分な SKU を用いて検証できること（CI パイプラインに含める）。

### 7.6 技術スタック（参照）

フロントエンド：`plan.txt` 記載の Next.js、React、TypeScript、Tailwind CSS、shadcn/ui。  
バックエンド：Supabase、PostgreSQL。

## 8. データ要件

### 8.1 エンティティ（論理）

以下は論理名の叩き台。物理テーブル名は実装で決定する。

| エンティティ | 説明 |
|--------------|------|
| profiles | 認証ユーザーと表示名・ロールの対応（物理名は `profiles` に統一。`auth.users.id` と 1:1）。 |
| product_groups | 親商品。 |
| product_skus | SKU。JAN、色、サイズ、現在庫数、再発注点、安全在庫、単価、画像参照など。 |
| inventory_movements | 在庫変動履歴。理由区分（手動、CSV、バーコード入庫、注文確定による減算等）、数量、日時、実行者（任意）。 |
| orders | 注文ヘッダ。注文番号、確定日時、税込合計、税額、税抜合計等。 |
| order_lines | 注文明細。SKU、数量、単価、行金額等。 |
| categories（任意） | カテゴリ。product_groups と関連付け可能。 |
| daily_sales_summary（任意） | 日次集計のキャッシュ。ビュー・クエリのみでも可。 |

### 8.2 整合性ルール

- 注文確定処理は **orders / order_lines の作成**と **SKU 在庫減算**および **inventory_movements 記録**を**同一 DB トランザクション**で行う（実装は `機能設計書`・`技術仕様書` に従い、**PostgreSQL の RPC（ストアドプロシージャ）**でまとめることを推奨）。
- 在庫不足時は確定不可（6.4.5）。

### 8.3 商品画像

- **Supabase Storage** に保存し、SKU にパスまたは公開 URL を保持する。

### 8.4 ダミーデータ（シード）

- **実行環境**：**開発環境（ローカル）のみ**。本番・Vercel では当該操作を**無効化または非表示**とする。
- **挙動**：**既存データの全削除（フルリセット）は行わない**。**追加（append）**を原則とし、自然キー（JAN、SKU コード、注文番号プレフィックス等）により**冪等**に近い投入とする。

## 10. スコープ外

- 決済（Stripe 等）。
- 注文のキャンセル・返品。
- バーコードによる**出庫**専用フロー。
- 税法レベルの税計算の正確性の保証。
- 高度な需要予測（ルールベースを超える統計・ML）。
- Google スプレッドシートを正とする双方向同期。
- 本番環境での全データ消去型シード。

---

## 11. 開発・移行フェーズ（推奨順）

1. Docker 上の PostgreSQL でスキーマ・マイグレーション方針を確立する。  
2. **早期に Supabase に移行**し、Auth・Storage・接続設定を本番想定で開発する。  
3. コア機能（認証、在庫、注文、ダッシュボード、エクスポート）を実装する。  
4. PWA、Sentry、Vercel Analytics、GitHub Actions・E2E・デプロイを整備する。  
5. 第2段階として RLS・一般ユーザー分離・監査ログを強化する。
