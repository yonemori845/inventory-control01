## 1. マイルストーンの分け方（なぜこの粒度か）

### 1.1 いくつに分けるべきか（提案）

**推奨：6 つのフェーズ（累積進捗の目安：約 20% → 40% → 55% → 75% → 90% → 100%）**

| 観点 | 説明 |
|------|------|
| **レビューしやすさ** | 各フェーズ末に「動くもの」または「検証可能な状態」を置き、ここで仕様の修正案を受け取れる。 |
| **依存関係** | DB スキーマ → 認証 → 在庫 → 注文（在庫減算）→ 周辺機能 → CI/CD の順が自然で、手戻りが減る。 |
| **負荷** | フェーズを細かくしすぎるとコンテキスト切替が増え、粗すぎるとレビュー単位が大きすぎる。**6 前後**がバランスが取りやすい。 |
| **無理のない中断** | フェーズ境界は「コミットして他作業に移っても破綻しにくい」タイミング（§2 参照）。 |

**補足**：ユーザー例の「20 / 40 / 50 / 80 / 100%」に近づけるため、本リストでは **55% と 75% の間に中間レビュー**を挟めるよう、**フェーズ C 末（55%）と D 末（75%）**を特に「指摘・修正」の推奨停止点とする（§2 の表）。

**細分化したい場合**：各フェーズ内のチェック項目を、**上半分／下半分**で区切れば、実質 **10 前後の停止点**にもできる（エージェントに「フェーズ B のチェックを半分まで」と指示可能）。

### 1.2 エージェント運用のコツ（Cursor）

- 作業開始時に **§3 の該当フェーズの「必読ドキュメント」**をコンテキストに含める。
- フェーズ完了時に **チェックをすべて `[x]` にする前に**、ユーザーが仕様変更を入れるなら、**対象の `doc/*.md` を先に直す**。
- **`要件定義書.md` の版数を上げたら**：下位設計書（機能・技術・デザイン・リポジトリ）のメタデータ「根拠」と改訂履歴を**セットで更新**する（ズレ防止）。

## 2. フェーズと累積進捗（目安）・中断のしやすさ

| フェーズ | 名称 | 累積進捗（目安） | ここで止めてもよい理由 |
|----------|------|------------------|-------------------------|
| **A** | プロジェクト基盤・DB スキーマ | **〜20%** | リポジトリと DB だけで「再現可能な土台」が完成。UI は最小でよい。 |
| **B** | 認証・レイアウトシェル | **〜40%** | ログイン後にナビ付きの空アプリが動く。以降は機能を縦に足すだけ。 |
| **C** | 在庫ドメイン | **〜55%** | マスタ・在庫・履歴・入庫スキャンまで一連で検証可能（注文は未着手）。 |
| **D** | 注文・ダッシュボード・レポート | **〜75%** | コア業務（注文確定＋在庫減）と可視化の主路線が通る。**仕様変更の指摘を入れやすい主停止点**。 |
| **E** | 設定・管理・シード・PWA・監視 | **〜90%** | 非機能・運用寄りをまとめ、本番相当の体験に近づける。 |
| **F** | CI/CD・E2E・デプロイ・仕上げ | **〜100%** | 課題の完成定義（要件 §2.3）に揃える。 |

**推奨レビュータイミング（例のパーセンテージに寄せた場合）**

- **〜20%** … スキーマ・環境の方向性  
- **〜40%** … 認証 UX・ナビ情報設計  
- **〜50%** … フェーズ C 中盤：在庫一覧・入れ子の見せ方（必要なら C を分割実装）  
- **〜80%** … フェーズ D 完了前後：注文フロー・税表示・エクスポート範囲  
- **〜100%** … CI・E2E・デプロイ確定後の最終確認
